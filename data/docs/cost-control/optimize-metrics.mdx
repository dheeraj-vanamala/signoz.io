This guide covers the primary causes of metric volume growth in SigNoz and how to address each one.

Metrics are measured in **Samples** — a single (timestamp, value) pair for one time series. Every unique combination of label values creates a distinct time series: a metric with a label carrying 500 unique values generates 500 distinct time series, each emitting a sample on every scrape interval.

👉 **[Understanding High Cardinality Data](https://signoz.io/blog/high-cardinality-data/)** — explains why cardinality multiplies time series and the downstream impact on storage and query performance.

## Signs Your Problem Is Metrics

- The **Cost Meter** shows a spike in `datapoint.count`.
- **Ingestion Analysis** shows one service dominating sample volume.

## Optimization Tiers

Address the problem as close to its source as possible before reaching for Collector processors:

1. **Instrumentation** — parameterize dynamic routes, remove debug metric emissions. Addresses the problem at source, before it enters the pipeline.
2. **OTel SDK** — use the [View API](https://signoz.io/opentelemetry/customize-metrics-streams-produced-by-opentelemetry-python-sdk/) to drop or rename attributes before they are serialized to OTLP. Per-language SDK environment variable controls are documented at [Drop OpenTelemetry Metrics](https://signoz.io/docs/userguide/drop-metrics/).
3. **OTel Collector** — use processors to aggregate or drop metrics in the pipeline. Covered in the problems below.

## Problem 1: Unused Metrics

The simplest reduction is to stop ingesting metrics that serve no purpose in any dashboard or alert. Drop them at the Collector before they reach SigNoz.

```yaml
processors:
  filter/drop_unused:
    error_mode: ignore
    metric_conditions:
      - 'IsMatch(metric.name, "^process\\.runtime\\..*")'
```

👉 **Read more:** [How to Drop OpenTelemetry Metrics](https://signoz.io/docs/userguide/drop-metrics/)

## Problem 2: High-Cardinality Labels

Kubernetes deployments commonly attach pod metadata to every metric via OTel Collector processors. If a label like `k8s.pod.name` has 200 unique values, it multiplies your baseline time series by 200×.

### Step 1: Find the Noisy Labels

1. Go to **Metrics > Metrics Explorer**. Toggle to the **Time Series** view.
2. Select your top ingested metric. Expand the **"All Attributes"** panel.
3. Identify labels with hundreds or thousands of unique values.

### Step 2: Verify Tooling Usage

Before aggregating anything, confirm the label is not in active use. **If it appears in any dashboard or alert, do not aggregate it — the aggregation will break those queries.**

- Check all Dashboard Panels for this label in `Group By` or `Filter` fields.
- Check all Alert conditions for references to this label.

### Step 3: Implement the Fix (Aggregation)

If the label is unused, aggregating it will reduce the number of time series to one per unique combination of the retained labels.

| Label | Why it's problematic | Verdict |
| --- | --- | --- |
| `k8s.pod.name` / `k8s.pod.uid` | Changes on every pod restart or redeploy. Large multiplier. | **Aggregate away** |
| `client.port` / `net.peer.port` | Ephemeral TCP ports (49152–65535). | **Aggregate away** |
| `url.full` / `http.target` | Dynamic query params. | **Aggregate away** |
| `db.query.text` | Full SQL with literal values. | **Aggregate away** |
| `http.route` | Unparameterized raw URLs (`/user/123` instead of `/user/{id}`). | **Fix at instrumentation** — parameterize route patterns in your SDK or framework router |

<Admonition type="warning">
Using `delete_key()` via the attributes processor **does NOT merge** time series. It removes the label name but leaves colliding, undefined data. You **must** use the `metricstransform` processor to aggregate across the dropped dimension.
</Admonition>

**Config Example:**

```yaml
processors:
  metricstransform/aggregate:
    transforms:
      - include: k8s\.pod\.network\.io
        match_type: regexp
        action: update
        operations:
          - action: aggregate_labels
            label_set: [k8s.namespace.name, direction] # Keeps only these labels, drops k8s.pod.name
            aggregation_type: sum
```

<Admonition type="warning">
`metricstransform` aggregates **within a single batch only** — it does not merge data points that arrive in separate batches or from separate exporter instances (e.g., multiple pods). If your high-cardinality series originate from multiple instances, this processor will produce partial, inconsistent results. Use the OTel SDK View API to drop the attribute at source instead.
</Admonition>

👉 **Read more:** [How to safely drop metric labels](https://signoz.io/docs/metrics-management/dropping-metric-labels/)

## Problem 3: Cumulative Temporality

By default, OpenTelemetry SDKs use **Cumulative Temporality** — they emit a sample carrying the running total on every scrape interval, even when the value is unchanged.

**Delta Temporality** only emits a sample when the value changes.

### When to Switch to Delta

If you have large idle infrastructure (e.g., 1000 pods with only 10 receiving traffic in a given minute), Delta prevents the 990 idle pods from emitting unchanged samples every scrape interval.

| Feature | Cumulative | Delta |
| --- | --- | --- |
| **Idle series** | Always sends | Skips unchanged |
| **Network failure** | Next scrape recovers the total (Safe) | Gap is permanently lost |
| **Collector memory** | Tracks running totals | Lower — no running totals to maintain |
| **UpDownCounter / Async UpDownCounter** | Cumulative | Cumulative (unaffected — Delta is undefined for bidirectional instruments) |

`OTEL_EXPORTER_OTLP_METRICS_TEMPORALITY_PREFERENCE=delta` applies Delta to `Counter`, `Async Counter`, and `Histogram` only. `UpDownCounter` and `Async UpDownCounter` remain Cumulative regardless.

<Admonition type="warning">
**Migration Warning:** Switching a live metric pipeline from Cumulative to Delta mid-flight may cause unexpected values in dashboards during the transition — panels computing rates or totals from these metrics may show incorrect readings until reporting stabilises. See [Metrics temporality FAQ](https://signoz.io/docs/metrics-management/troubleshooting/faqs/) for details on temporality mismatch behaviour.
</Admonition>

**How to Set Delta:**

```bash
export OTEL_EXPORTER_OTLP_METRICS_TEMPORALITY_PREFERENCE=delta
```

## Validate Your Changes

<Admonition type="note">
The Cost Meter Dashboard uses `signoz.meter.*` metrics which aggregate hourly — changes may take up to one hour to appear there. Use the Metrics Explorer for faster feedback first.
</Admonition>

To confirm the volume reduction:

1. Navigate to **Metrics > Metrics Explorer**. In the list view, verify that unused metrics you dropped no longer appear, or that the time series count for your target metric has reduced.
2. Wait 1 hour, then check the **Cost Meter Dashboard** filtered by your target service and confirm the `datapoint.count` trend has dropped.
