This guide covers the primary approaches to reducing trace ingestion volume in SigNoz.

Trace volume is measured in **Gigabytes (GB)** ingested. High-frequency, low-latency successful requests — health checks, background polling, routine CRUD operations — generate disproportionate span volume relative to their diagnostic utility.

Identify which span operations are generating the highest volume, assess whether they have operational value, and apply the appropriate solution below.

## Solution 1: Head-Based Sampling

Head-based sampling makes the keep/drop decision at trace creation time in the SDK, before any span data is serialized or transmitted. It requires no Collector memory and no gateway Collector.

**Trade-off:** The sampling decision is made before the trace completes, so error traces have the same drop probability as successful ones — you cannot guarantee 100% error trace retention. If that is a requirement, use tail-based sampling instead.

Configure the `ParentBased(TraceIdRatioBased)` sampler via environment variable:

```bash
export OTEL_TRACES_SAMPLER=parentbased_traceidratio
export OTEL_TRACES_SAMPLER_ARG=0.1  # Retain 10% of all traces
```

<Admonition type="note">
`parentbased_traceidratio` is the correct sampler for distributed systems — it respects parent span sampling decisions so the entire trace is kept or dropped together. Using plain `traceidratio` makes an independent per-span decision and produces broken partial traces when requests cross service boundaries.
</Admonition>

**[Sampling strategies and trade-offs](https://signoz.io/blog/saving-money-with-sampling-strategies/)** — covers ParentBased(TraceIdRatioBased), reservoir sampling, adaptive sampling, and byte-rate limiting.

## Solution 2: Tail-Based Sampling

Tail sampling buffers the complete trace in Collector memory before making a sampling decision. This allows you to retain traces based on their final outcome — errors, high latency — regardless of the sampling rate applied to healthy traces.

**Standard Configuration:**

```yaml
processors:
  tail_sampling:
    decision_wait: 10s
    num_traces: 50000
    expected_new_traces_per_sec: 1000
    policies:
      # Keep 100% of errors
      - name: errors
        type: status_code
        status_code: { status_codes: [ERROR] }

      # Keep 100% of slow traces (>1s)
      - name: slow-traces
        type: latency
        latency: { threshold_ms: 1000 }

      # Retain 5% of all remaining healthy traces
      - name: baseline
        type: probabilistic
        probabilistic: { sampling_percentage: 5 }
```

<Admonition type="warning">
**Memory and Topology:** `num_traces: 50000` holds up to 50,000 in-flight traces in Collector memory simultaneously. Always pair `tail_sampling` with a [`memory_limiter` processor](https://signoz.io/docs/opentelemetry-collection-agents/opentelemetry-collector/configuration/) to prevent OOM crashes. Tail sampling requires all spans from a given trace to arrive at the same Collector instance — this means it must run on a dedicated gateway Collector that receives all traffic, not on per-pod sidecar Collectors. See the <a href="https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/processor/tailsamplingprocessor" target="_blank" rel="noopener noreferrer nofollow">OpenTelemetry tail sampling processor documentation</a> for deployment topology guidance.
</Admonition>

## Solution 3: Pre-Filter Before Sampling

<Admonition type="note">
Dropping a span at the Collector does not automatically remove custom metrics generated from that span via the `spanmetrics` processor. Metric volume is managed independently — see [Optimize Metrics](https://signoz.io/docs/cost-control/optimize-metrics/).
</Admonition>

Pre-filtering known zero-value spans before the tail sampler is a recommended practice regardless of traffic volume. Spans that are definitively not useful should never enter sampler memory — this reduces the evaluation surface and improves sampler efficiency.

Two common drop patterns based on what you found in Trace Explorer:

**By span operation name** (the `name` field surfaced in Step 4 of the analysis guide):

```yaml
processors:
  filter/drop_noise:
    error_mode: ignore
    traces:
      span:
        - 'name == "grpc.health.v1.Health/Check"'
        - 'IsMatch(name, "GET /internal/.*")'
```

**By URL path attribute** (for HTTP spans where the operation name alone is too broad):

```yaml
processors:
  filter/drop_noise:
    error_mode: ignore
    traces:
      span:
        - 'IsMatch(attributes["url.path"], ".*health.*")'
        # Adjust the pattern to match your actual endpoint paths before deploying
```

Both patterns can be combined in a single `filter/drop_noise` processor block. Pipeline ordering is critical — `filter/drop_noise` must execute before `tail_sampling`:

```yaml
service:
  pipelines:
    traces:
      processors: [filter/drop_noise, tail_sampling]
```

**Read more:** [How to configure Drop Spans & Tail Sampling](https://signoz.io/docs/traces-management/guides/drop-spans/)

## Validate Your Changes

<Admonition type="note">
The Cost Meter Dashboard uses `signoz.meter.*` metrics which aggregate hourly — changes may take up to one hour to appear there. Use the Trace Explorer for faster feedback.
</Admonition>

To confirm the volume reduction:

1. Navigate to **Traces**, swap to the **Table** tab. Filter by your target service, group by `name` (span operation name), and sort by count — verify the high-volume operations you targeted have reduced.
2. Wait 1 hour, then check the **Cost Meter Dashboard** filtered by your target service and confirm the span GB trend has dropped.
