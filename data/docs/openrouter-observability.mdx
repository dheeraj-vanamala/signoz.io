## What is OpenRouter Observability?

OpenRouter observability gives you full visibility into your LLM API routing across multiple models. This guide walks you through setting up OpenRouter monitoring using <a href="https://opentelemetry.io/" target="_blank" rel="noopener noreferrer nofollow">OpenTelemetry</a> and exporting traces to SigNoz, so you can track requests, latency, and usage across your OpenRouter applications.

With full OpenRouter observability in SigNoz, you can correlate traces across your AI and LLM workflows, configure alerts on latency and errors, and gain actionable insights to continuously improve reliability and responsiveness. SigNoz brings your OpenRouter telemetry into unified dashboards so you can monitor every model request in one place.

## Prerequisites

- A [SigNoz Cloud account](https://signoz.io/teams/) with an active ingestion key or [Self Hosted SigNoz instance](https://signoz.io/docs/install/self-host/)
- Internet access to access OpenRouter API and send telemetry data to SigNoz Cloud
- An <a href="https://openrouter.ai/" target="_blank" rel="noopener noreferrer nofollow">OpenRouter account</a> and OpenRouter API key. You can get it from the <a href="https://openrouter.ai/settings/keys" target="_blank" rel="noopener noreferrer nofollow">OpenRouter API Keys page</a>.


## Set Up OpenRouter Observability with OpenTelemetry

### Step 1: Configure broadcasting traces from OpenRouter to OTel Collector

For more information on setting up traces from OpenRouter refer to the <a href="https://openrouter.ai/docs/guides/features/broadcast/otel-collector" target="_blank" rel="noopener noreferrer nofollow">OpenRouter broadcast guide</a>.

#### Enable Broadcast in OpenRouter

Go to <a href="https://openrouter.ai/settings/observability" target="_blank" rel="noopener noreferrer nofollow">Settings > Observability</a> and toggle Enable Broadcast.

<Figure
  src="/img/docs/llm/openrouter/openrouter-enable-broadcast.webp"
  alt="OpenRouter Enable Broadcast Toggle"
  caption="OpenRouter Enable Broadcast Toggle"
/>

#### Configure OpenTelemetry Collector

Click 'Add Destination +' next to the OpenTelemetry Collector entry

<Figure
  src="/img/docs/llm/openrouter/openrouter-add-destination.webp"
  alt="OTel Collector Add Destination"
  caption="OpenTelemetry Collector Add Destination"
/>



and enter:
- **Endpoint**: `https://ingest.<region>.signoz.cloud:443/v1/traces`
- **Headers**:
```json
{
  "signoz-ingestion-key": "<your-ingestion-key>"
}
```
<Figure
  src="/img/docs/llm/openrouter/openrouter-add-details.webp"
  alt="OTel Collector Add Details"
  caption="OpenTelemetry Collector Add Details"
/>

- `<region>`: Your [SigNoz Cloud region](https://signoz.io/docs/ingestion/signoz-cloud/overview/#endpoint)
- `<your-ingestion-key>`: Your SigNoz [ingestion key](https://signoz.io/docs/ingestion/signoz-cloud/keys/)


<Admonition type="info">
  For self-hosted collector:

  1. Deploy your OpenTelemetry Collector with an OTLP receiver
  2. Configure the receiver to listen on a publicly accessible endpoint
  3. Add the endpoint URL
  4. Remove the ingestion key header as shown in [Cloud → Self-Hosted](https://signoz.io/docs/ingestion/cloud-vs-self-hosted/#cloud-to-self-hosted).
</Admonition>

#### Test and Save

Click Test Connection to verify the setup. Once there is a success message, click the "Add" button.

Once configured, your OpenRouter calls are traced and sent to SigNoz.

## Troubleshooting OpenRouter Observability
</ToggleHeading>

If you don't see your telemetry data:

1. **Verify network connectivity** - Ensure your application can reach OpenRouter API and SigNoz Cloud endpoints
2. **Check ingestion key** - Verify your SigNoz ingestion key is correct
3. **Wait for data** - OpenTelemetry batches data before sending, so wait 10-30 seconds after making API calls

### What is the OpenTelemetry Collector?

Think of the OTel Collector as a middleman between your app and SigNoz. Instead of your application sending data directly to SigNoz, it sends everything to the Collector first, which then forwards it along.

### Why use it?

- **Cleaning up data** — Filter out noisy traces you don't care about, or remove sensitive info before it leaves your servers.
- **Keeping your app lightweight** — Let the Collector handle batching, retries, and compression instead of your application code.
- **Adding context automatically** — The Collector can tag your data with useful info like which Kubernetes pod or cloud region it came from.
- **Future flexibility** — Want to send data to multiple backends later? The Collector makes that easy without changing your app.

See [Switch from direct export to Collector](https://signoz.io/docs/opentelemetry-collection-agents/opentelemetry-collector/switch-to-collector/) for step-by-step instructions to convert your setup.

For more details, see [Why use the OpenTelemetry Collector?](https://signoz.io/docs/opentelemetry-collection-agents/opentelemetry-collector/why-to-use-collector/) and the [Collector configuration guide](https://signoz.io/docs/opentelemetry-collection-agents/opentelemetry-collector/configuration/).
