OpenTelemetry is a Cloud Native Computing Foundation(CNCF) project aimed at standardizing the way we [instrument applications](https://signoz.io/docs/instrumentation/) for generating telemetry data(logs, metrics, and traces).<br />
However, OpenTelemetry does not provide storage and visualization for the collected telemetry data. For doing so, you need an OpenTelemetry-compatible UI. The data collected from OpenTelemetry-instrumented applications can be sent to a backend of your choice, which can then be visualized.

OpenTelemetry is quietly becoming the web standard for instrumenting cloud-native applications, which are often based on distributed systems with complex communication patterns. This is especially true in 2026 where applications must also juggle [LLM API calls, rate limits, etc.](https://signoz.io/blog/llm-observability/), beyond just their own business logic. <br />

There are a few reasons why OpenTelemetry is the preferred choice for generating telemetry data:

- **Lightweight:**
  OpenTelemetry APIs are lightweight and are completely decoupled from any implementation. By default, OpenTelemetry APIs are <Tooltip text="no-op implementations" content="Implementations that don't do anything when called (No Operation)" />.
- **Diverse:**
  OpenTelemetry provides instrumentation in almost all the major programming languages. It also covers instrumentation for a large number of open-source libraries and frameworks.
- **Community:**
  OpenTelemetry is incubating under CNCF, is backed by a huge community and major cloud vendors. The huge community support makes it a safe choice for future-proofing your observability layer.

Once telemetry data is collected with OpenTelemetry, it needs to be stored and visualized for end-user consumption. A good visualization of telemetry data collected by OpenTelemetry can lead to quicker insights to debug performance issues.

A few charts possible with OpenTelemetry data can be:

- Application performance metrics charts

Before we learn more about visualizations in an OpenTelemetry UI, let’s have a brief overview of OpenTelemetry.

## What is OpenTelemetry?

[OpenTelemetry](https://signoz.io/opentelemetry/) is an open-source collection of tools, APIs, and SDKs that aims to standardize the way we generate and collect telemetry data. It was formed after the merger of two open-source projects, OpenCensus and OpenTracing, in 2019.

It follows a specification-driven development: the <a href = "https://github.com/open-telemetry/opentelemetry-specification" rel="noopener noreferrer nofollow" target="_blank">OpenTelemetry specification</a> has design and implementation guidelines for how the instrumentation libraries should be implemented.

The specification is designed into distinct types of telemetry known as signals. Presently, OpenTelemetry has specifications for these three signals:

- Logs
- Metrics and
- Traces

Together, these three signals form the [three pillars of observability](https://signoz.io/blog/three-pillars-of-observability/). OpenTelemetry is the bedrock for setting up an observability framework. The application code is instrumented using OpenTelemetry client libraries, which enables the generation of telemetry data. Once the telemetry data is generated and collected, you need to configure the OpenTelemetry exporter to send data to a backed analysis tool like [SigNoz](https://signoz.io/).

## Why do we need an OpenTelemetry UI?

OpenTelemetry UI serves as a powerful tool for monitoring, analyzing, and optimizing the performance and behavior of your applications and infrastructure - all via a user-friendly interface. This, in turn, helps in making informed decisions based on real-time and historical data.

Here are some key benefits of using an OpenTelemetry UI:

- **Debugging and troubleshooting:** With an OpenTelemetry UI, identifying errors and performance bottlenecks in distributed applications becomes swift. <br />For instance, it aids in pinpointing which service(s) might be experiencing delays or encountering errors, streamlining the debugging and troubleshooting process to uncover the root cause.
- **Monitoring and observability:** An OpenTelemetry UI allows for [continuous monitoring](https://signoz.io/comparisons/continuous-monitoring-tools/) of the performance and overall health of distributed applications. This empowers engineers to observe trends in performance metrics, error rates, and other vital indicators over time.
- **Analytics and reporting:** Beyond monitoring and observability, an OpenTelemetry UI also help identify trends and patterns that might not be immediately obvious. <br />For example, engineers can choose to find and optimize the "hot paths" used by applications based on past 30 days' data.
- **Alerting:** An OpenTelemetry UI provides the ability to set alerts for specific events, such as when performance thresholds are exceeded or errors occur. In the event of a critical service outage, these alerts promptly notify engineers, enabling them to take immediate action.
## What kind of visualization should an OpenTelemetry UI include?

An effective OpenTelemetry UI should incorporate a range of visualizations tailored to provide comprehensive insights into the performance and behavior of applications and infrastructure.

Here are some key types of visualizations that should be included:

### APM metrics

APM metrics are essential for understanding how applications perform in real-world scenarios. Some key APM metrics are:

- Latency - p90, p99, p50 latencies of a service
- Application throughput
- Error rates
- Apdex

The ideal OpenTelemetry UI should include APM metrics to provide a comprehensive view of application performance. [APM metrics](https://signoz.io/docs/userguide/metrics/#what-are-application-metrics) mentioned above can be calculated from trace data collected by OpenTelemetry.

### Distributed Tracing

Distributed tracing is a powerful observability feature for tracking the journey of a request as it flows through the various components of a distributed system. It provides a comprehensive view of user requests across various services and network boundaries. It can help identify performance bottlenecks and optimize system performance.

In an OpenTelemetry UI, you should be able to do the following with [OpenTelemetry trace](https://signoz.io/blog/opentelemetry-distributed-tracing-part-1/) data:

**Switch easily from application metrics to corresponding trace data.**

This allows you to investigate trace data around timestamps where APM metrics are not performing well.

<Figure src="/img/blog/common/metrics_correlation.webp" alt="Access traces and logs from metrics for a given timestamp" caption="Access traces and logs from metrics for a given timestamp" />

**List view of all spans**

This allows you to sort through spans based on duration. The duration can be relative, such as last 30 minutes, or an absolute time range. You can also apply filters to only fetch spans matching certain criteria.

<Figure src="/img/docs/instrumentation/golang/trace-list.webp" alt="List view of spans has all the spans around a particular timestamp" caption="List view of spans has all the spans around a particular timestamp" />

**Charts for aggregates on trace data**

This is a powerful visualization available in some OpenTelemetry UI like SigNoz. You can apply aggregates like `avg`, `p90`, `p99`, etc. and group it by things like `service.name`, `http.method`, `status.code` , etc.

### Flamegraphs & Gantt Charts

Flamegraphs and Gantt charts are two powerful visualization features that can be used to analyze OpenTelemetry data. Flamegraphs can be used to visualize the flow of requests in a trace. Gantt charts provide a timeline view of task execution, helping to identify resource contention and bottlenecks.

### Trace Exploration with Queries

Trace exploration with queries refers to the capability within an observability tool, to search and analyze traces (sequences of events) based on specific criteria or conditions.

A good OpenTelemetry UI empowers developers to perform targeted investigations. They can achieve this by applying filters or conditions to the traces, allowing them to narrow down their search. This focused approach enables them to concentrate on specific transactions, requests, or events within a distributed system.

<Figure src="/img/blog/common/trace_aggregates.webp" alt="Trace Explorer provided in an OpenTelemetry UI like SigNoz helps you dig deeper in trace data" caption="Trace Explorer provided in an OpenTelemetry UI like SigNoz helps you dig deeper in trace data" />

In production environments, trace exploration with queries proves invaluable. For instance, during a service outage, they become instrumental in troubleshooting or deriving near real-time, actionable insights and understanding application behaviour patterns.

### Metrics Dashboard

A metrics dashboard displays a collection of key performance indicators (KPIs) and metrics relevant to the system's health and performance. This can include metrics related to resource utilization (e.g. CPU, memory), network traffic, and application-specific metrics.

An OpenTelemetry UI should include visualization for a metrics dashboard, like host metrics and Kubernetes metrics dashboard.

- **Host Metrics Dashboard**
This dashboard provides a specialized view of performance indicators specific to the underlying host system. It captures crucial data related to the utilization of system resources, including CPU, memory utilization, disk usage, and network activity. These metrics are instrumental in fine-tuning resource allocation and uncovering possible performance constraints.

### Logs

Logs provide a detailed record of events and activities within an application or system, providing in-depth insights into events, errors, and transactions, enabling effective troubleshooting and analysis.

An OpenTelemetry UI should include a logs visualization for the following benefits:

- **Proactive [System Health Monitoring](https://signoz.io/guides/server-health-monitoring/):** Through continuous [log monitoring](https://signoz.io/blog/log-monitoring/), potential system vulnerabilities and impending issues can be identified and addressed before they escalate into full-blown outages. This preemptive approach is crucial for maintaining uninterrupted service.
- **Centralized Log View:** A centralized view of logs from all components of a distributed system streamlines the process of issue identification and troubleshooting. This consolidated perspective offers a comprehensive overview of system activities.
- **Dynamic Filtering and Rapid Search Capabilities:** These capabilities allow developers to efficiently sift through logs, to precisely locate specific information even within large and extensive log datasets.
- **Visual Data Representation:** Visual representation aids in the identification of trends, patterns, and anomalies within the log data, enabling more detailed analysis.

## FAQs

### Does OpenTelemetry have a UI?
No, OpenTelemetry itself doesn't provide a UI. OpenTelemetry is a collection of tools, APIs, and SDKs for generating and collecting telemetry data. For visualization, you need to use a separate backend and UI tool, such as SigNoz, Jaeger, or Zipkin.

### What is the purpose of OpenTelemetry?
The main purpose of OpenTelemetry is to standardize the way telemetry data (metrics, traces, and logs) is generated, collected, and transmitted in distributed systems. It aims to make instrumentation consistent across various languages and frameworks, enabling better observability in complex, cloud-native environments.

### What is telemetry in frontend?
In frontend development, telemetry refers to the collection of data about how users interact with a web application. This can include performance metrics (like page load times), user behavior data (like click patterns), and error reporting. Frontend telemetry helps developers understand user experience and application performance from the client-side perspective.

### What is the difference between OpenTelemetry and Prometheus?
OpenTelemetry and Prometheus serve different purposes in the [observability stack](https://signoz.io/guides/observability-stack/):
- OpenTelemetry is a standard for generating, collecting, and exporting telemetry data (metrics, traces, and logs).
- Prometheus is primarily a metrics collection and alerting system. It uses a pull-based model to scrape metrics from targets.

While OpenTelemetry can be used to collect metrics that Prometheus can scrape, it also handles traces and logs, which Prometheus does not.

### What is the difference between Jaeger and OpenTelemetry collector?
- Jaeger is a distributed tracing system that can be used as a backend for storing and visualizing trace data.
- The OpenTelemetry Collector is a vendor-agnostic component that can receive, process, and export telemetry data to various backends, including Jaeger.

The OpenTelemetry Collector is more versatile as it can handle metrics and logs in addition to traces, and can send data to multiple backends simultaneously.

### Who is behind OpenTelemetry?
OpenTelemetry is a Cloud Native Computing Foundation (CNCF) project. It was formed by merging the OpenCensus (from Google) and OpenTracing (from Uber) projects. It's now supported by a large community of contributors from various companies including Google, Microsoft, Lightstep, Splunk, and many others.

### Is OpenTelemetry push or pull?
OpenTelemetry supports both push and pull models, depending on the signal type and configuration:
- For traces and logs, it typically uses a push model.
- For metrics, it can support both push and pull models, depending on the exporter configuration.

### When to use OpenTelemetry?
You should consider using OpenTelemetry:
1. When building or maintaining distributed systems or microservices architectures.
2. When you need a standardized way to instrument your applications across different languages and frameworks.
3. When you want to future-proof your observability strategy and avoid vendor lock-in.
4. When you need comprehensive observability data (metrics, traces, and logs) from your applications.

### What are the cons of OpenTelemetry?
While OpenTelemetry offers many benefits, some potential drawbacks include:
1. Learning curve: It can be complex to set up and configure, especially for large systems.
2. Overhead: Instrumentation can add some performance overhead, though it's generally minimal.
3. Evolving standard: As it's still evolving, there might be changes or inconsistencies across different language implementations.
4. Requires additional tools: OpenTelemetry doesn't provide storage or visualization, so you need to integrate with other tools.

### What is the difference between telemetry and OpenTelemetry?
- Telemetry is a general term referring to the collection of measurements or other data at remote points and their automatic transmission to receiving equipment for monitoring.
- OpenTelemetry is a specific, open-source framework and standard for generating, collecting, and exporting telemetry data. It provides a standardized way to implement telemetry in software systems.

### Is OpenTelemetry free?
Yes, OpenTelemetry is free and open-source. It's released under the Apache 2.0 license. However, some backends or visualization tools you might use with OpenTelemetry may have associated costs.

### What is the main purpose of telemetry?
The main purpose of telemetry is to collect data about the performance, usage, and behavior of systems or applications, often from remote or inaccessible locations. This data is used for monitoring, troubleshooting, optimization, and making informed decisions about system management and development.

### What is the difference between OpenTelemetry and OpenTracing?
OpenTracing was one of the precursor projects to OpenTelemetry, focusing specifically on distributed tracing. OpenTelemetry is a more comprehensive project that:
1. Covers metrics and logs in addition to traces.
2. Provides a more complete set of tools and APIs for instrumentation.
3. Is actively developed and supported by the CNCF, while OpenTracing is now in maintenance mode.

### How to visualize OpenTelemetry data?
To visualize OpenTelemetry data:
1. Choose a compatible backend system (e.g., SigNoz, Jaeger, Zipkin, Prometheus).
2. Configure your OpenTelemetry Collector or SDK to export data to this backend.
3. Use the UI provided by your chosen backend to view and analyze the data.
4. Alternatively, you can use general-purpose visualization tools like Grafana, which can connect to various data sources.

### Which of the following are features of OpenTelemetry?
Key features of OpenTelemetry include:
1. Support for multiple languages and frameworks
2. Standardized data collection for metrics, traces, and logs
3. Vendor-agnostic design
4. Automatic instrumentation capabilities
5. Flexible data export options
6. Support for custom instrumentation
7. Sampling and data processing capabilities
